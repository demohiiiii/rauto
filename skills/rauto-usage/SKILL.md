# Rauto Usage

Execute rauto operations directly for users whenever possible.
Do not default to tutorial-style answers.

## Core Mode

Prefer action-first behavior:

1. Parse user goal and classify it as install/setup, read-only, or config-changing.
2. For install/setup requests, inspect the environment first and prefer official GitHub Releases binaries unless the user explicitly wants a source build.
3. For read-only operations, run the relevant `rauto` command immediately.
4. For config-changing operations, prefer `tx`/`tx-workflow`/`orchestrate` with rollback-aware planning.
5. If commands are generated by the agent, ask for human confirmation before execution.
6. Return key results (not raw noise), plus command used.

## Execution Rules

1. For install/setup/upgrade requests:
   - inspect current state first: `command -v rauto`, `rauto --version`, current OS/arch, and whether the current repo is a `rauto` checkout.
   - choose install path from user intent:
     - default: official GitHub Releases binary for the current platform
     - fallback: current repo / local dev build: `cargo build --release`
     - fallback: install current checkout only when the user explicitly wants source-based install
   - verify after install with resolved binary path and `rauto --version`.
2. For read/query requests, execute immediately:
   - examples: `device list`, `templates list`, `show-connection`, `connection-history`, `replay --list`.
3. For config-changing requests, prefer rollback-capable execution:
   - single block: `tx`
   - multi-step/multi-resource: `tx-workflow`
   - multi-device staged rollout: `orchestrate`
4. For generated change commands, do not execute immediately:
   - show planned command(s), rollback strategy, and require user confirmation.
5. For explicit read-only commands (for example `show`), tx/workflow is not required.
6. Resolve connection in this priority:
   - explicit command args > `--connection <name>` > ask for missing fields.
7. Do not ask the user to manually run commands if agent can run them.
8. Summarize outputs with important fields:
   - target, mode, success/failure, key errors, next action.

## Risk Guardrails

Require explicit user confirmation before destructive actions:

- upgrade/reinstall that may overwrite an existing `rauto` install when the target is ambiguous
- any binary replacement into shared/system paths such as `/usr/local/bin`
- `rauto backup restore ... --replace`
- profile/template/connection delete operations
- tx/workflow execution that changes config when user intent is ambiguous

Additional safety requirements for config changes:

- Prefer user-local install paths with GitHub Releases binaries; do not silently replace another installed copy in shared/system paths.
- Use source builds only when the user explicitly wants this checkout or when release download is unavailable.
- Prefer `tx`/`tx-workflow`/`orchestrate` with rollback-aware planning over direct apply patterns.
- Run/offer `--dry-run` first for workflow-based changes.
- Never auto-run agent-generated change commands without confirmation.
- Include rollback path in the proposal before execution.

If user explicitly asks to execute destructive action, proceed.

## Missing Input Strategy

Ask only for missing must-have fields:

- For install/setup:
  - need desired install target only when ambiguous: GitHub Releases binary, repo-local build, or install current checkout from source.
- For `exec/template/tx/tx-workflow/orchestrate/test-connection`:
  - need either full host credentials or usable `--connection`.
- For `replay`:
  - need record file path or JSONL source.
- For history queries:
  - need connection name.

## Response Format

When command is executed, report:

1. `Operation`: what was run
2. `Command`: exact rauto command
3. `Result`: key output summary
4. `Notes`: risk, errors, or follow-up actions
